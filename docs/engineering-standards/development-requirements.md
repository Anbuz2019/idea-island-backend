# 项目开发要求与工程规范

本文档用于沉淀当前项目在代码风格、分层架构、设计边界、测试方式、日志要求和变更约束上的统一标准，作为后续开发、重构、Review 和验收的长期基准。

## 1. 总体原则

1. 先看文档，再动代码。
   开发前必须先阅读顶层 `design.md`、`docs/design/requirements.md` 以及相关模块说明文档。
2. 以“需求一致 + 测试通过”作为验收标准。
   不能只看代码是否存在，必须同时满足需求文档、设计文档和测试结果。
3. 发现问题先沉淀成文档。
   Gap report、checklist、复查清单等统一放入 `docs/`。
4. 重构必须服务于边界清晰。
   不做只有形式、没有实际收益的拆分。

## 2. 架构与分层要求

### 2.1 DDD 分层边界

1. `trigger` 层只负责对外触发与适配。
   包括 Controller 实现、Filter、Listener、Job、上下文透传和入参到领域命令的转换。
2. `trigger` 层不得承载领域业务逻辑。
   业务规则、认证规则、状态流转、标签规则等必须放在 `domain`。
3. `domain` 层负责实体、值对象、领域服务、仓储抽象和业务规则。
4. `infrastructure` 层负责数据库、Redis、MQ、COS 等外部依赖实现。
5. 领域访问基础设施时，必须通过仓储接口或适配器接口，不能让 `trigger` 直接操作底层实现。

### 2.2 API 模块职责

1. 所有 HTTP Controller 契约统一放在 `api` 模块。
2. `trigger` 层的 Controller 只保留实现类，不再持有接口定义。
3. 每个 Controller 对应的 `request` 和 `response` DTO 统一放在 `api` 模块。
4. `trigger` 层可以负责 DTO 与领域对象之间的映射，但 DTO 的定义归 `api` 所有。
5. 不允许在 `trigger` 模块重复声明一份与 `api` 等价的请求响应对象。

### 2.3 Domain Service 组织方式

1. 所有 `domain` 层 service 必须先定义接口。
2. 接口命名统一为 `IxxxService`。
3. 接口统一放在：
   `domain/<context>/service/IxxxService.java`
4. 具体实现类名称保持业务语义不变，例如 `UserService`、`MaterialService`。
5. 实现类统一放在：
   `domain/<context>/service/impl/XxxService.java`
6. 所有依赖注入统一依赖接口，不直接依赖实现类。

### 2.4 依赖方向

1. 上层不得依赖下层实现类。
   `trigger` 只能依赖 `domain` 接口和 `api` 契约，不能依赖 `domain.service.impl`。
2. Controller 对外暴露的契约对象、请求对象、返回对象统一由 `api` 模块提供。
3. 无用代码、空转配置、未使用依赖必须成链路清理，不能只删表层文件。

## 3. 代码设计要求

### 3.1 业务规则设计

1. 业务实现必须与需求文档保持一致。
2. 并发正确性不能依赖“先查后写”。
   例如 `existsByEmail + save` 只能做友好提示，正确性必须依赖数据库约束或最终一致性机制。
3. 底层异常必须翻译为明确的业务异常。
   例如唯一键冲突应转换为业务冲突，而不是直接返回 500。
4. 安全链路必须闭环。
   包括 JWT 生成、Redis token 校验、续签、登出失效等。
5. 自动化流程必须可追踪。
   如资料提交后的异步处理、系统标签刷新、自动失效等，不允许无痕运行。

### 3.2 命名与结构

1. 接口使用 `I` 前缀，例如 `IUserService`、`IMaterialService`。
2. 实现类保留业务语义名称，例如 `UserService`、`TopicService`、`StatusTransitionService`。
3. 目录结构应直接体现接口、实现、仓储抽象、基础设施实现的职责边界。
4. `api` 模块中的 Controller 接口建议与实现类一一对应，例如：
   `IAuthController` 对应 `AuthController`。

## 4. 日志要求

1. `trigger` 层必须记录请求侧关键日志。
   至少覆盖接口入口、写操作成功、校验失败、业务异常。
2. `domain` 层必须记录业务侧关键日志。
   至少覆盖实体创建、状态流转、规则拒绝、并发冲突回退、自动补全等关键链路。
3. 日志分级统一如下：
   成功写操作使用 `info`
   业务拒绝、鉴权失败、参数非法使用 `warn`
   系统异常、外部依赖异常使用 `error`
   高频只读查询使用 `debug`
4. 日志必须带可定位字段。
   根据场景至少包含 `userId`、`topicId`、`materialId`、动作名、状态变化等信息。
5. 日志不得输出敏感信息。
   禁止直接打印密码、验证码、token、完整手机号、完整邮箱等；必要时必须脱敏。
6. 关键日志必须优先服务线上排障，而不是仅服务本地开发。

## 5. 测试要求

### 5.1 验收标准

1. 默认以 `mvn -q test` 通过作为最低验收门槛。
2. 模块完整性检查时，以“功能存在 + 测试覆盖 + 测试通过”共同判断，而不是只看实现代码。

### 5.2 测试风格

1. 单元测试采用 BDD 风格组织。
   推荐 `@Nested + @DisplayName`，测试命名采用 `given... when... then...` 语义。
2. 单元测试断言优先使用 AssertJ 链式写法，提高可读性。
3. HTTP 层测试可以使用 `MockMvc`，但行为断言必须清晰表达预期。
4. `MockMvc` / `WebMvc` 这类集成测试统一放在 `app` 层测试目录，不放在 `trigger` 层。
4. 修复缺陷时，必须补对应测试，包括正常路径、边界路径、错误路径和安全路径。

### 5.3 测试边界

1. 未经确认，不得为了测试方便擅自修改数据库结构。
2. 即便是测试 `schema`，也属于结构变更，原则上需要先说明。
3. 如果只是覆盖异常路径，优先使用 mock、stub 或异常桩，不优先改表结构。

## 6. 文档要求

1. 模块检查后必须产出文档沉淀。
   可包括 gap report、模块清单、分层说明、测试清单、复查结论等。
2. 代码变更后，相关文档必须同步更新，不能长期滞后。
3. 模块介绍文档至少包含：
   模块目标
   分层设计
   核心链路
   主要接口
   测试清单
   后续复查点

## 7. 变更约束

1. 不要擅自修改数据库结构。
   包括生产 DDL、测试 schema、表约束等。
2. 如确需修改结构，必须先说明原因、影响范围和验证方式。
3. 清理无用代码时，要清理整条链路。
   包括源码、配置、属性类、依赖和对应文档。
4. 所有结构性调整都必须经过编译和测试验证。

## 8. 当前明确要求清单

以下要求已在项目沟通中明确提出，视为必须遵守：

1. 开发前先阅读需求文档和设计文档。
2. 缺陷分析结果必须形成文档并放入 `docs/`。
3. 功能检查必须对照需求文档和设计文档。
4. 模块完整性评估以测试通过和测试覆盖为标准。
5. 单元测试使用 BDD 风格。
6. 单元测试断言优先使用 AssertJ 链式写法。
7. 严格遵守 DDD 分层，不在 `trigger` 层放业务 service。
8. `domain` 层 service 必须抽接口。
9. `domain` 层实现统一放到 `service/impl`。
10. 所有注入位置统一依赖 `IxxxService`。
11. 所有 Controller 接口统一放在 `api` 模块。
12. 所有 Controller 的 `request` 和 `response` 统一放在 `api` 模块。
13. `trigger` 层 Controller 只负责实现 `api` 契约和做适配转换。
14. 未经确认不要修改数据库结构，包括测试 schema。
15. 并发问题按数据库约束与异常翻译方式修复，不只靠前置查询。
16. 无用代码和无效依赖要及时清理。
17. 重构后必须重新执行编译和测试。
18. `trigger` 层和 `domain` 层关键位置必须具备用于线上排障的日志。

## 9. 推荐执行顺序

后续新增模块或重构模块时，建议按以下顺序执行：

1. 阅读需求与设计文档。
2. 输出模块现状 gap 文档。
3. 修正架构边界与实现问题。
4. 按 BDD 风格补齐测试。
5. 运行全量测试。
6. 更新模块说明文档与复查清单。
